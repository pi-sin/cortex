# Skill: Product Spec

Before building any feature, write a 1-page spec covering:

## The Problem (2-3 sentences)

What user pain does this solve? Be specific — "it's annoying" isn't a pain point.
"Users spend 12 min/day manually copying Slack decisions into Jira comments" is.

## The Solution (2-3 sentences)

What does Cortex do about it? Describe the experience, not the implementation.

## User Stories (3-5)

"As a [type of user], I want to [action], so that [outcome]."

## Success Metrics (2-3 measurable outcomes)

- Leading indicator (e.g., "60% of users interact with widget in first week")
- Lagging indicator (e.g., "self-reported time saved increases by 5 min/day")

## Scope

- In Scope: what we're building
- Out of Scope: what we're explicitly NOT building (often more important)
- Future: things we might add later

## Edge Cases (3-5)

What could go wrong or confuse users?

## Validation Rule

Every feature must connect to one of the 5 core pain points:

1. Fragmented context
2. Notification overload
3. Repetitive cross-tool workflows
4. Context-switching cost
5. Meeting overhead

If it doesn't map, question whether to build it at all.
